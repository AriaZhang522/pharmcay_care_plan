# 拿到新项目后，怎么判断「哪些要看、哪些是业务逻辑」

这是一套通用思路，不限于 Django，下次换一个项目也可以按这个顺序来。

---

## 第一步：先认「项目类型」和「入口」

看根目录和文档，快速确认三件事：

1. **技术栈**：后端是 Django / Flask / Node / Java …？前端是 React / Vue / 纯静态？
2. **入口在哪**：哪个命令或哪个服务是「启动整个应用」的？（例如 `docker compose up`、`npm run dev`、`python manage.py runserver`）
3. **有没有 README / 架构图**：有的话先扫一遍，知道「大概分几块、数据从哪到哪」

这样你就知道：**请求从哪进来、最后到哪**，后面找业务逻辑会快很多。

---

## 第二步：按「层次」分文件，而不是按文件名猜

不管什么框架，大多数项目都可以分成这几层。先对号入座，再决定看谁：

| 层次 | 一般是啥 | 典型位置/名字 | 要不要重点看 |
|------|----------|--------------|--------------|
| **入口 / 路由** | 把「URL 或请求」分发给具体处理函数 | `urls.py`、`routes/`、`router.js`、`App.jsx` 里的路由 | ✅ 要看：用来找「一个请求会进哪个文件、哪个函数」 |
| **业务逻辑（核心）** | 真正做「这个项目在干嘛」的代码 | `views.py`、`services/`、`api/`、`handlers/`、组件里调接口 + 处理数据的部分 | ✅✅ 最要看：改功能、修 bug、做需求都在这 |
| **数据层** | 和数据库/存储打交道的定义和读写 | `models.py`、`schema/`、`entities/`、ORM/DAO 文件 | ✅ 要看：和表结构、查询、存储相关的改动都在这里 |
| **配置** | 环境、连接、开关、常量 | `settings.py`、`.env.example`、`config/`、`constants.js` | ⚪ 扫一眼知道「从哪读配置」即可，改配置时再细看 |
| **框架/脚手架** | 框架约定、启动脚本、生成器 | `manage.py`、`migrations/`、`next.config.js`、各种 `__init__.py` | ❌ 一般不细看，当「系统给的」 |
| **构建/部署** | 怎么装依赖、怎么打镜像、怎么跑 | `Dockerfile`、`docker-compose.yml`、`package.json` 的 scripts、CI 配置 | ⚪ 部署或改依赖时再看 |

**简单记：**  
先找 **入口/路由** → 顺着找到 **业务逻辑** 和 **数据层**，这三类是你「必须能讲清楚」的；**配置**和**构建**知道在哪、干什么就行；**框架/脚手架**默认当黑盒，用到再查文档。

---

## 第三步：用「请求流」反推业务逻辑在哪

目标是：**一个具体操作（例如「提交表单」「点某个按钮」）从请求发出到写库/调外部 API，经过了哪些文件、哪些函数？**

1. **从前端找触发点**  
   在页面/组件里搜：按钮的 `onClick`、`submit`、`fetch`/`axios` 的 URL。  
   → 得到：**请求的 URL 和可能带的参数**。

2. **在后端找「谁处理这个 URL」**  
   在项目里搜这个 URL（例如 `/api/generate-care-plan/`）。  
   → 找到：**路由/路由表**（例如 `urls.py`、`routes.ts`）。  
   → 路由会指向一个 **处理函数/Controller**（例如 `views.generate_care_plan`）。  
   → 那个函数所在的文件，就是**这一条业务逻辑所在的地方**。

3. **顺着调用链往下看**  
   在处理函数里看：它调了哪些**别的函数、别的模块**（例如 `call_llm_for_care_plan`、`Order.objects.create`）。  
   → 被调用的如果是「和业务强相关」的（算价格、生成 Care Plan、校验规则），就在**业务逻辑**里；  
   → 如果是「查库、写库、调外部 API」，就分别属于**数据层**或**外部服务封装**。  
   这样你就能画出一条线：**入口 → 业务函数 → 数据/外部调用**，这条线上的文件都是「和这个功能直接相关」的，需要看。

4. **数据从哪来、写到哪**  
   看业务逻辑里用到的「模型/表/实体」和「读写的函数」在哪个目录（例如 `models.py`、`repositories/`）。  
   → 这些就是**数据层**，改表结构、改查询、改存储时要看。

按这个流程走一两遍，你就能标出：**这个项目里，和「某个具体功能」相关的文件是哪些**，它们就是你要重点看的；其它要么是配置，要么是框架/构建，按需再看。

---

## 第四步：用「命名和目录」做快速判断（经验法则）

很多项目会按约定命名，可以用来辅助判断，而不是盲目每个文件都打开：

- **名字里带 `test`、`spec`、`mock`、`fixture`** → 测试/假数据，审业务逻辑时可以先不看，除非你在改测试。
- **名字叫 `config`、`settings`、`constants`、`.env`** → 配置，知道在哪、干什么就行。
- **`migrations`、`schema`（仅表结构定义）、`__init__.py`（空或只做 import）** → 框架/结构用，一般不逐行审。
- **`utils`、`helpers`、`lib`** → 可能是通用小工具，被业务逻辑调用；**谁调了它们**决定你要不要看（被核心流程调到的就看一下）。
- **`api/`、`services/`、`handlers/`、`views/`** → 很大概率是**业务逻辑或接口层**，优先看。
- **`models/`、`entities/`、`repositories/`、`dao/`** → 数据层，和表/查询相关的要看。

**注意：** 命名只是提示，不保证 100% 准确；最终以「这个文件在请求流里有没有被用到、是不是在算业务规则/写库」为准。

---

## 第五步：列一个「本项目的文件清单」（可选但推荐）

第一次理清之后，可以给自己写一张小表，以后审代码或 onboard 别人都能用：

- **入口/路由**：例如 `app/urls.py`、前端的路由组件。
- **业务逻辑**：例如 `app/views.py`、`app/services/xxx.py`。
- **数据层**：例如 `app/models.py`。
- **配置**：例如 `app/settings.py`、`.env.example`。
- **不用审的**：例如 `manage.py`、`migrations/`、`__init__.py`。

你之前的 **BACKEND_FILES_GUIDE.md** 其实就是这个项目后端的一份「文件清单」；新项目也可以照这个格式写一页，放在项目里或你的笔记里。

---

## 总结成一张「检查清单」

拿到新项目时，可以按顺序问自己：

1. 这个项目是啥技术栈？启动命令是啥？有没有 README/架构说明？
2. 请求从哪进来？（前端路由 / 后端路由）→ 找到**入口**。
3. 我关心的这个功能，对应哪个 URL / 哪个入口？→ 找到**处理这个功能的文件/函数**。
4. 这个函数又调了哪些别的模块？→ 画出**业务逻辑 + 数据层**的调用链。
5. 数据层、配置、构建、测试、migrations 分别在哪些目录？→ 给它们贴上「重点看 / 扫一眼 / 不用看」的标签。

这样你就不会「不知道从哪看起」，也能判断：**哪个文件要看、哪个不用看、哪个文件里是业务逻辑**。  
下次拿到新项目，从「认类型 + 找入口」开始，再按「请求流」跟一条完整链路，就能快速把「要看」和「不用看」分开。
